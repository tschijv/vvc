Opmerkingen n.a.v. mail Peter Makkes

1. Naamswijziging

De applicatie heet nu "GEMMA Softwarecatalogus" en moet hernoemd worden naar "Voorzieningencatalogus". Dit raakt layout, metadata, OG-image, en alle referenties in de code.

2. Rollenmodel

Peter beschrijft drie hoofdrollen:

  • Aanbod-beheerder (leveranciers) — geïmplementeerd als LEVERANCIER rol
  • Gebruik-beheerder (gemeenten/samenwerkingen) — geïmplementeerd als GEMEENTE_BEHEERDER
  • Gebruik-raadpleger — geïmplementeerd als GEMEENTE_RAADPLEGER

De huidige implementatie matcht redelijk maar mist het zelf-registratie en fiatteringsproces.

3. Autorisatie

  • ✅ Gemeenten zien alles inclusief elkaars landschappen
  • ✅ Leveranciers zien alleen hun eigen aanbod en gebruik
  • ✅ Bezoekers zien alleen openbare info

4. Suite-concept

Peter geeft aan dat het suite-concept voorlopig is losgelaten. Geen actie nodig.

5. GEMMA Views

Peter verwijst naar GEMMA Archi repository view. De kaart-functionaliteit (/kaart) biedt al GEMMA views, maar de specifieke view-structuur zou gevalideerd moeten worden.

Conclusies

✅ Sterke punten

  1. Kernfunctionaliteit aanbod & gebruik — Pakketten, leveranciers, gemeenten, standaarden volledig uitgewerkt
  2. GEMMA-integratie — Views, glossary, AMEFF export/import
  3. Autorisatiemodel — 10 rollen, correct filteren per rol
  4. API — Publieke REST API met OpenAPI docs
  5. Vergelijkfunctie — Side-by-side gemeente-vergelijking
  6. Fuzzy search — Typfouten-tolerant (pg_trgm)
  7. Responsiveness & Dark mode

❌ Belangrijkste gaps

  1. Zelf-registratie & fiattering — Alles gaat via admin
  2. 2FA/TOTP — Alleen email + wachtwoord
  3. Multi-organisatie toegang — User = 1 organisatie
  4. Organisatie-samenvoegen — Essentieel bij herindeling
  5. Geautomatiseerde tests — Geen tests aanwezig
  6. OTAP / CI/CD — Geen pipeline of omgevingen
  7. Webstatistieken — Geen Matomo of vergelijkbaar
  8. E-mailfunctionaliteit — Geen wachtwoord-reset/notificaties
IBD-wensen: volledig open terrein
Alle IBD-gerelateerde wensen (BIO compliance, DPIA, pen-tests, DigiD, kwetsbaarheden, register van verwerkingen) zijn niet gerealiseerd. Dit is een groot functioneel domein dat aanzienlijke ontwikkeltijd vergt en deels afhankelijk is van GEMMA-uitbreidingen.

Aanbeveling prioritering

De huidige applicatie dekt de kernfunctionaliteit goed af. De grootste gaps zitten in:

  1. Gebruikersbeheer-workflows — zelf-registratie, fiattering, multi-org
  2. Beveiligingseisen — 2FA, audit logging
  3. DevOps — testen, CI/CD, OTAP
  4. IBD/compliance domein — volledig nieuw te bouwen

Prioritering zou moeten liggen bij de harde eisen die nog niet gerealiseerd zijn, met name de gebruikersbeheer-workflows en beveiligingseisen, aangezien deze randvoorwaardelijk zijn voor productie-gebruik.
